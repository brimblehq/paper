---
title: "Run Docker inside a sandbox"
description: "Enable Docker during sandbox creation and run nested containers from your SDK."
---

Pass `docker: true` when creating a sandbox. Brimble installs Docker, starts the daemon, and marks the sandbox `ready` only after `docker info` succeeds — no manual bootstrap scripts.

## When to use this recipe

- Spinning up Postgres, Redis, or other services with `docker compose` during a test run.
- Letting an agent build and run containers without host Docker access.
- Prototyping a multi-service setup in a disposable environment.

## Prerequisites

- A Brimble account with sandbox access and `BRIMBLE_SANDBOX_KEY` set.
- The SDK installed (see [SDKs](../sdks#install)).
- Outbound network for pulling packages and images (`egress.mode` defaults to `open`).
- A Debian/Ubuntu template such as `node-22` or `python-3.12`.
- **1 GB+ RAM** recommended (`docker: true` defaults memory to 1024 MB if you omit `specs.memory`).

## Recipe

Create a sandbox with `docker: true`, then run containers.

<CodeGroup>

```typescript TypeScript
import { Sandbox } from "@brimble/sandbox";

const client = new Sandbox();

const handle = await client.sandboxes.createReady({
  region: "auto",
  template: "node-22",
  docker: true,
  specs: { cpu: 1000, memory: 1024 },
});

try {
  const hello = await handle.exec({
    cmd: "docker run --rm hello-world",
    timeout_seconds: 300,
  });
  console.log(hello.stdout);

  await handle.exec({
    cmd: "docker run -d --rm --network=host --name demo nginx:alpine",
    timeout_seconds: 300,
  });
} finally {
  await handle.destroy();
}
```

```python Python
from brimble_sandbox import Sandbox

client = Sandbox()

handle = client.sandboxes.create_ready({
    "region": "auto",
    "template": "node-22",
    "docker": True,
    "specs": {"cpu": 1000, "memory": 1024},
})

try:
    hello = handle.exec({
        "cmd": "docker run --rm hello-world",
        "timeout_seconds": 300,
    })
    print(hello["stdout"])
finally:
    handle.destroy()
```

```go Go
package main

import (
	"context"
	"fmt"

	sandbox "github.com/brimblehq/brimble-sdks/sandbox-go"
)

func main() {
	ctx := context.Background()
	client, _ := sandbox.NewClient(sandbox.ClientConfig{})

	handle, _ := client.Sandboxes.CreateReady(ctx, sandbox.CreateSandboxRequest{
		Region:   "auto",
		Template: "node-22",
		Docker:   true,
		Specs:    &sandbox.SandboxSpecs{CPU: 1000, Memory: 1024},
	}, nil)
	defer handle.Destroy(ctx)

	result, _ := handle.Exec(ctx, sandbox.ExecInput{
		Cmd:            "docker run --rm hello-world",
		TimeoutSeconds: 300,
	})
	fmt.Println(result.Stdout)
}
```

</CodeGroup>

## What's happening

1. **`docker: true`.** During provisioning, Brimble runs `apt-get install docker.io`, uploads a startup script, and starts `dockerd` before the sandbox becomes `ready`.
2. **`createReady`.** Waits until Docker is installed and the daemon responds to `docker info`.
3. **`docker run`.** Talks to the local daemon over `/var/run/docker.sock` inside the sandbox.
4. **`--network=host`.** Port mapping via `-p` is not supported inside sandboxes — bind on the host network instead.

On a [persistent sandbox](persistent-dev-env) with `docker: true`, Docker is reinstalled automatically when you resume from pause.

## Limitations

- **Port mapping:** use `--network=host`, not `-p` / `--expose`.
- **Install time:** first boot spends a few minutes installing packages; the sandbox stays in `starting` until Docker is ready.
- **Performance:** nested containers are slower than running Docker on a bare VM.
